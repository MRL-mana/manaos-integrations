# Phase2 自己観察 設計メモ

**前提**: Phase1 で継続率・満足度・同一テーマ再訪（ユニーク数）を計測済み。
**最短で動かす**: [自己観察 クイックスタート](SELF_OBSERVATION_QUICKSTART.md) を参照。

---

## Phase2 の狙い

1. **テーマ再訪の「頻度」**
   Phase1 は「複数スレッドで出現したテーマの個数」。Phase2 では「同じテーマが何回スレッドで出現したか」の分布を取る。

2. **振り返りの活用**
   満足度・理由をログに蓄積し、低満足度ターンの傾向分析やプロンプト改善に使う。

3. **第三層の検討**
   知識グラフ・ベクトルDB との連携は、テーマ再訪の「何を覚えて再利用するか」の設計次第。

## 指標案

| 指標 | Phase1 | Phase2 |
|------|--------|--------|
| テーマ再訪 | ユニーク数（何テーマが2+スレッドで出現） | 頻度分布（テーマごとの出現回数） |
| 満足度 | 平均・条件別 | 低満足度ターンの理由クラスタリング |
| 継続率 | そのまま | そのまま |

## 知識グラフ/ベクトルDB 検討メモ（要件固め用）

- **目的**: テーマ再訪時に「過去の会話で得た知見」を参照するか、単なるテーマ出現頻度で止めるか。
- **検討項目**
  - 格納する単位: テーマID、満足度・理由、スニペットのどれを入れるか
  - 参照タイミング: 応答前（プロンプトに注入）か、振り返り時のみか
  - 既存システムとの役割: System3 RAG・学習システムとの重複を避けるため、自己観察専用の「振り返りメモ」に限定するか
- **結論**: 要件が固まり次第、候補（例: ローカルベクトルDB、Neo4j 等）を比較して PoC を検討

## 第三層 PoC の範囲（推奨デフォルト）

- **目的**: 同一テーマ再訪時に「過去の振り返りメモ」を参照できるようにする。まずはベクトル検索は使わず、テーマID（1ターン目ユーザー発話の上位3語）でキー検索する最小構成で検証する。
- **格納単位**: テーマID（`word1/word2/word3`）、thread_id、turn_id、satisfaction、reason、timestamp。自己観察専用の「振り返りメモ」に限定し、System3 RAG とは役割を分ける。
- **参照タイミング**: PoC では「参照 API を用意するだけ」とし、応答前のプロンプト注入は次のイテレで検討。
- **実装**: `phase2_reflection_memo.py` で JSONL にメモを追加・テーマIDで取得。`phase2_backfill_memos.py` で既存の phase1 ログからメモを投入。将来的にベクトルDB／知識グラフへ差し替え可能なインターフェースにする。
- **応答前プロンプト注入**: `PHASE2_MEMO_INJECT=on` のとき、`/api/llm/chat` で同一テーマの過去振り返りメモを「最後のユーザー発話の先頭」に付与する。メモは「満足度X: 理由...」形式で最大5件。無効時は従来どおり応答のみ。
- **リアルタイム追記**: `PHASE2_MEMO_APPEND=on` のとき、振り返りを記録するたびに `phase2_reflection_memos.jsonl` に追記する。バックフィル不要でメモが蓄積され、次回同一テーマで注入される。
