# Phase 1 (Read-only) Go/No-Go 判定表

## 📋 判定フロー

Phase 1開始後、以下の2点を共有してください：

1. **SECURITYログ1行**
2. **ダッシュボード初期値（数字）**

これらに基づいて即座に判定します。

---

## 🔒 SECURITYログ判定

### ✅ Go（必須条件）

以下のすべてが満たされていること：

- `auth=enabled`
- `rate_limit=enabled`
- `max_input` が想定どおり（例：200000）
- `pii_mask=enabled`

**例**:
```
SECURITY: auth=enabled, rate_limit=enabled, max_input=200000, pii_mask=enabled
```

### ❌ No-Go（即停止/修正）

以下のいずれかが該当する場合：

- 1つでも `disabled` が混ざる
- `max_input` が極端に小さい（例：5000とか）
- ログ自体が出ない（適用されてない疑い）

**修正方法**:
- `.env`ファイルを確認
- 環境変数が正しく読み込まれているか確認
- サービスを再起動

---

## 📊 ダッシュボード初期値判定

### 1) E2E p95

#### ✅ Go
- 普段の想定内で安定（目安：< 0.3秒、環境次第）
- スパイクが連発しない

#### ⚠️ 注意
- 急に跳ねる
- スパイクが連発

#### ❌ No-Go
- 連続して高止まり（前回比2倍以上が続く）
- 秒単位になったら「別のボトルネック（RAG/IO/LLM本体/ログ爆増）」が濃厚

**判定基準**:
- ベンチマーク結果（p95=0.005秒）を基準に、0.1秒以下ならGo
- 0.1秒〜0.3秒なら注意
- 0.3秒以上ならNo-Go

---

### 2) ゲート遮断率

#### ✅ Go
- 0〜80%の範囲で推移

#### ⚠️ 注意
- 80〜95%が継続（参照がほぼ使われてない）

#### ❌ No-Go
- 95〜100%が継続（"常に信用されないメモリ"＝無意味 or 汚染判定過敏）

**判定基準**:
- 0〜80%: Go
- 80〜95%: 注意（監視を強化）
- 95〜100%: No-Go（即停止）

---

### 3) 矛盾検出率

#### ✅ Go
- 低い（< 5%）
- または安定（急増しない）

#### ⚠️ 注意
- じわじわ上がる（キー設計 or TTL/隔離の運用確認）

#### ❌ No-Go
- 急増（前日比2倍以上）
- 常時高い（> 10%）（入力ノイズ源 or 判定ロジック過敏）

**判定基準**:
- < 5%: Go
- 5〜10%: 注意（監視を強化）
- > 10% or 急増: No-Go（即停止）

---

### 4) スロット使用率（偏りチェックの代理指標）

#### ✅ Go
- 過度な偏りがない（急に特定に寄る動きがない）
- 分散が適切（mean_variance < 100程度）

#### ⚠️ 注意
- 偏り傾向が見える（ロードバランサ/キー生成を疑う）

#### ❌ No-Go
- 極端な集中（"崩壊の前兆"）
- 分散が異常に大きい（mean_variance > 1000）

**判定基準**:
- mean_variance < 100: Go
- 100〜1000: 注意（監視を強化）
- > 1000: No-Go（即停止）

---

### 5) 書き込み回数/分（Read-onlyの重要チェック）

#### ✅ Go
- 0（または限りなく0）

#### ❌ No-Go
- 0じゃない
  → Read-onlyなのに書いてる＝**設定ミス**。即修正。

**判定基準**:
- 0: Go
- > 0: No-Go（即修正）

**修正方法**:
- `FWPKM_WRITE_MODE=readonly`を確認
- `FWPKM_WRITE_ENABLED=0`を確認
- サービスを再起動

---

## 🚀 Phase 2（Write 10%）へ進むための進行条件

Phase 1を **24〜48時間** 回して、以下が満たされればPhase 2へ。

### Phase 2 Go条件（確定版）

- [ ] **SECURITYログが毎回正しい**（起動のたびにOK）
- [ ] **重大エラー（5xx）が継続発生してない**
- [ ] **Read-onlyなのに書き込みが発生してない**（書き込み回数/分 = 0）
- [ ] **p95が安定**（急増なし、0.1秒以下で安定）
- [ ] **矛盾検出率が安定**（急増なし、< 5%で安定）
- [ ] **ゲート遮断率が95%貼り付きじゃない**（0〜80%の範囲で推移）

### Phase 2 No-Go条件

以下のいずれかが該当する場合、Phase 2へ進まない：

- SECURITYログが正しくない
- 重大エラー（5xx）が継続発生している
- Read-onlyなのに書き込みが発生している
- p95が急増している（前日比2倍以上）
- 矛盾検出率が急増している（前日比2倍以上）
- ゲート遮断率が95%貼り付き

## 📝 実行のコツ

### Phase 1の監視スケジュール

- **最初の1時間**: 監視を濃く（10分おきにダッシュボード見る）
- **問題がなければ**: 以降は1時間〜数時間おきでOK
- **異常が出たら**: まず **Kill Switch**（止めてから考えるのが正解）

### 判定のタイミング

- **起動直後**: SECURITYログ + ダッシュボード初期値で判定
- **24時間後**: ダッシュボードで指標を確認、Phase 2 Go条件をチェック
- **48時間後**: 最終判定、Phase 2へ進むか決定

---

## 🔄 判定結果に基づくアクション

### Go判定の場合

1. Phase 1を継続（24〜48時間）
2. 定期的にダッシュボードを確認
3. Phase 2 Go条件を満たしたらPhase 2へ

### 注意判定の場合

1. 監視を強化（30分おきにダッシュボードを確認）
2. ログを確認
3. 問題が解決したらGo判定に変更
4. 問題が継続する場合はNo-Go判定に変更

### No-Go判定の場合

1. **即座にKill Switch**（`export FWPKM_WRITE_ENABLED=0`）
2. ログを確認
3. 設定を修正
4. 修正後、再度Phase 1を開始

## 📊 判定表（簡易版）

| 項目 | Go | 注意 | No-Go |
|------|----|----|-------|
| SECURITYログ | すべてenabled | - | 1つでもdisabled or ログなし |
| E2E p95 | < 0.1秒 | 0.1〜0.3秒 | > 0.3秒 or 急増 |
| ゲート遮断率 | 0〜80% | 80〜95% | 95〜100% |
| 矛盾検出率 | < 5% | 5〜10% | > 10% or 急増 |
| スロット使用率 | 分散 < 100 | 100〜1000 | 分散 > 1000 |
| 書き込み回数/分 | 0 | - | > 0 |

---

## 🎯 判定の流れ

1. **SECURITYログを確認**
   - Go: すべてenabled → 次へ
   - No-Go: 1つでもdisabled → 即停止・修正

2. **ダッシュボード初期値を確認**
   - 各項目を判定表に照らし合わせて判定
   - すべてGo → Phase 1継続
   - 1つでもNo-Go → 即停止・修正
   - 注意がある → 監視を強化

3. **24〜48時間後、Phase 2 Go条件をチェック**
   - すべて満たしている → Phase 2へ
   - 1つでも満たしていない → Phase 1継続 or 修正
