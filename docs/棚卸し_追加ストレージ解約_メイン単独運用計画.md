# 追加ストレージ解約 → メイン単独運用計画

解約後は **メインストレージ (99G) のみ** で運用します。

---

## 1. 現状の容量

| ストレージ | サイズ | 使用中 | 空き |
|------------|--------|--------|------|
| **メイン** (/) | 99G | 52G | **43G** |
| **追加** (/mnt/storage500) | 492G | 42G | 428G → **解約で消失** |

⚠️ **注意**: 追加の 42G（Docker等）をメインの空き 43G に収める必要あり。Trinity/OpenWebUI は母艦移行済み。

---

## 2. 追加ストレージの内訳（移行が必要なもの）

| 項目 | サイズ | 用途 |
|------|--------|------|
| docker | 19G | コンテナイメージ・overlay2 |
| docker-data | 4.7G | Docker データ |
| trinity_workspace | 4.7G | Trinity tasks.db 等 |
| docker-volumes | 403M | PostgreSQL, Weaviate, MinIO 等 |
| mongodb | 157M | MongoDB |
| postgres | 70M | PostgreSQL |
| langfuse-postgres | 46M | Langfuse |
| その他 | 〜1M | openwebui, node_exporter 等 |
| **合計** | **〜29G** | （docker+docker-data の重複要確認） |

※ 実測では storage500 全体で 47G。Docker の data-root が storage500 を参照している場合、docker/docker-data/docker-volumes が一体で使われている可能性あり。

---

## 3. 対応オプション

### A. メインの空きを増やす（推奨）

| 施策 | 見込み削減 |
|------|------------|
| /root 整理（残り11G） | 5〜8G |
| /var/log ローテーション強化 | 1G |
| snap 不要パッケージ削除 | 1〜2G |
| **合計** | **7〜11G** |

→ メイン空き 43G + 7G ≒ **50G** 確保できれば移行可能。

### B. 一部サービスを母艦へ移行

| 候補 | サイズ | 備考 |
|------|--------|------|
| Trinity | 4.7G | 母艦で稼働させる場合 |
| Open WebUI | 448K | 小さいが母艦に移行も可 |
| MongoDB | 157M | 使っていなければ停止 |

### C. 一部サービスを停止

| 候補 | サイズ | 影響 |
|------|--------|------|
| Langfuse | 46M | 学習分析用。不要なら停止 |
| node_exporter | 20M | メトリクス。監視やめれば停止 |
| InfluxDB | - | メトリクスDB。要確認 |

---

## 4. 推奨移行手順

### Phase 1: 解約前の準備（メイン側の整理）

1. /root の不要ファイル・DB を削除（目標: 11G → 5G）
2. /var/log を journalctl + logrotate で 500MB 以下に維持
3. snap の不要パッケージ・旧リビジョン削除

### Phase 2: データ移行

1. **Docker の data-root をメインへ変更**
   - `/etc/docker/daemon.json` で `data-root: "/var/lib/docker"` などに変更
   - 既存データは `rsync` 等で storage500 → メインへコピー

2. **Trinity のワークスペース移行**
   - サービス停止
   - `rsync -av /mnt/storage500/trinity_workspace/ /root/trinity_workspace/`
   - 起動設定（docker-compose 等）のパスをメインに変更
   - 再起動

3. **その他ボリューム**
   mongodb, postgres 等は Docker volume 経由なら、data-root 移行に含まれる可能性あり。別 volume の場合は同様にパス変更。

### Phase 3: 解約・検証

1. 追加ストレージ解約（VPS のディスク detach 等）
2. 全サービス起動確認
3. ディスク使用量の監視

---

## 5. 解約前チェックリスト

- [ ] メイン空きを 50G 以上に増やした
- [ ] Docker data-root をメインに移行済み
- [ ] Trinity ワークスペースをメインに移行済み
- [ ] 全コンテナがメインのパスを参照している
- [ ] n8n ワークフローは母艦にエクスポート済み
- [ ] 必要データのバックアップ完了（母艦 or クラウド）

---

## 6. 注意点

- **Docker**: data-root 変更後は `systemctl restart docker` が必要。コンテナは再作成の可能性あり。
- **Trinity**: tasks.db は大切。移行前にバックアップ推奨。
- **解約のタイミング**: 移行と検証が完了してから解約すること。
