# v1.1 訓練「6日コース」の切り分けと短縮設定

## 事実確認（いまの設定）

| 項目 | 値 | 備考 |
|------|-----|------|
| **eval** | **オフ** | `--no-eval` 指定 → `eval_strategy="no"`。評価は学習中に走っていない。 |
| **save_steps** | **100** | 100 step ごとにチェックポイント保存。4325 step なら **43 回保存**。 |
| **save_total_limit** | 3 | 最新 3 個のみ保持。 |
| **GPU** | **使用中** | ログに `GPUメモリ使用量: 14.24GB / 15.92GB`。CUDA 使用。 |
| **dataloader_num_workers** | 0 | Windows 互換のため。データローダは単一スレッド。 |
| **max_length** | 2048 | デフォルト。短くすると学習は速くなる。 |
| **disable_tqdm** | True | 進捗バーオフのため、ログに it/s は出ない。 |

## 結論：遅い主因は「保存頻度」

- **243 分 / 100 step ≒ 2.43 分/step** → 純粋な 1 step が 2 分超は現実的には重い。
- eval はオフなので、**100 step あたりの時間の多くは「チェックポイント保存」**の可能性が高い。
- 100 step ごとに数 GB をディスクに書き、43 回やると I/O でかなり時間が溶ける。

## 短縮のための変更（優先順）

### 1) save_steps を大きくする（効果：大）

- **100 → 500 または 1000** に変更。
- 保存回数: 43 回 → **約 9 回 or 約 5 回**。
- 同じ「純粋学習時間」なら、**数日分短縮**が期待できる。

**やり方**: `run_v11_train_fast.bat` を使う（`--save-steps 500` 済み）。
または手動で:

```bash
python castle_ex\train_castle_ex_full.py ... --save-steps 500 ...
```

### 2) v1.1 検証は「短い step で勝敗を見る」（効果：大）

- 目的は「Layer2 が跳ねたか」の確認なので、**1000〜2000 step で一度止めて評価**で十分。
- フル 4325 step は本番用。検証は `--max-steps 2000` で打ち切る。

**やり方**:

```bash
python castle_ex\train_castle_ex_full.py ... --max-steps 2000 --save-steps 500 ...
```

### 3) max_length を下げる（効果：中〜大、必要なら）

- 2048 → 1024 や 512 にすると step あたり時間がかなり短くなる。
- castle_ex の典型タスクなら 512〜1024 で足りることも多い。

```bash
... --max-length 1024
```

### 4) 1 step の秒数 / it/s を確認したい場合

- 現在は `disable_tqdm=True` のためログに it/s は出ない。
- TensorBoard の `logs/` に step ごとの loss は出ているので、**step 間の時間差**から逆算可能。
- 必要なら `logging_steps` ごとに経過秒数を print する処理をスクリプトに追加すれば、ログで「○秒/step」を確認できる。

## いまからやること（推奨）

1. **いま動いている訓練を止める**（すでに止めているならそのままでOK）。
2. **短縮用で再開**: `run_v11_train_fast.bat` を実行（`--save-steps 500`、25 epoch のまま）。
3. まずは **検証だけ** したい場合は、同じバッチの「検証用」ブロックを有効にして `--max-steps 2000` で回す。
4. 終わったら `castle_ex_evaluator_fixed.py` で評価し、Layer2 の invalid=0 / acc を確認。

これで「6日」が **数時間〜1日程度** に収まる想定。
