## 🐳 コンテナ関連の問題点・改善点

### 🔴 緊急対応が必要な問題

#### 1. リソース制限の欠如 ⚠️

**問題**: `docker-compose.manaos-services.yml`にメモリ・CPU制限が設定されていない

**現状**:
- すべてのサービスにリソース制限なし
- リソース競合の可能性
- 他のプロセスへの影響が不明

**影響**:
- メモリ不足によるシステムクラッシュ
- CPU使用率の暴走
- 他のサービスのパフォーマンス低下

**対策**:
```yaml
# 改善例
services:
  unified-api:
    deploy:
      resources:
        limits:
          cpus: '2.0'
          memory: 2G
        reservations:
          cpus: '0.5'
          memory: 512M
```

**優先度**: 🔴 高

---

#### 2. サービス依存関係の不明確さ ⚠️

**問題**: `depends_on`が設定されていないため、サービス間の依存関係が不明確

**現状**:
- 起動順序が保証されていない
- 依存サービスが起動していない状態で起動する可能性
- エラーの原因特定が困難

**影響**:
- 起動時のエラー
- 接続エラーの多発
- サービス間の連携不良

**対策**:
```yaml
# 改善例
services:
  mcp-api:
    depends_on:
      unified-api:
        condition: service_healthy
      llm-routing-api:
        condition: service_healthy
```

**優先度**: 🔴 高

---

#### 3. Healthcheckの問題 ⚠️

**問題**: Healthcheckで`curl`コマンドを使用しているが、curlがインストールされているか不明

**現状**:
- すべてのサービスで`curl`を使用
- Dockerfileでcurlがインストールされているか不明確
- Healthcheckが失敗する可能性

**影響**:
- Healthcheckが正しく動作しない
- サービス状態の正確な監視ができない
- `depends_on`の`condition: service_healthy`が機能しない

**対策**:
```dockerfile
# Dockerfileでcurlを確実にインストール
RUN apt-get update && apt-get install -y \
    curl \
    && rm -rf /var/lib/apt/lists/*
```

または、Pythonの`requests`ライブラリを使用したヘルスチェックスクリプトを作成

**優先度**: 🔴 中

---

### 🟡 中程度の問題点

#### 4. Windowsパスの直接指定 ⚠️

**問題**: 環境変数でWindowsパスが直接指定されている

**現状**:
```yaml
environment:
  - OBSIDIAN_VAULT_PATH=C:/Users/mana4/Documents/Obsidian Vault
```

**影響**:
- Linux環境で動作しない
- ポータビリティの低下
- コンテナ内外のパス整合性の問題

**対策**:
- 環境変数でパスを管理し、デフォルト値を設定
- ボリュームマウントでパスを統一

**優先度**: 🟡 中

---

#### 5. ネットワーク設定の最適化 ⚠️

**問題**: `extra_hosts`が設定されているが、サービス間の依存関係が明示されていない

**現状**:
- `host.docker.internal`を使用してホストマシンにアクセス
- サービス間の通信が明示されていない

**影響**:
- ネットワークパフォーマンスの低下
- デバッグが困難

**対策**:
- サービス間通信はDockerネットワークを使用
- `extra_hosts`は外部サービス接続時のみ使用

**優先度**: 🟡 低

---

#### 6. ログ管理の不足 ⚠️

**問題**: ログのローテーション設定がない

**現状**:
- ログが無制限に増加する可能性
- ディスクスペースの消費

**影響**:
- ディスク不足
- ログ検索の困難

**対策**:
```yaml
services:
  unified-api:
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"
```

**優先度**: 🟡 低

## 🔌 MCPサーバー関連の問題点・改善点

### 🔴 緊急対応が必要な問題

#### 1. MCPサーバーの分散・統合不足 ⚠️

**問題**: 複数のMCPサーバーが存在し、統合が不完全

**現状**:
- `n8n_mcp_server` - n8nワークフロー管理
- `svi_mcp_server` - SVI動画生成
- `manaos_unified_mcp_server` - 統合MCPサーバー
- その他、複数のMCPサーバーが存在

**影響**:
- CursorのMCP設定が複雑
- 機能の重複
- メンテナンスの困難

**対策**:
1. `manaos_unified_mcp_server`への統合を推奨
2. 個別MCPサーバーは必要な場合のみ使用
3. 統合MCPサーバーの機能拡張

**優先度**: 🔴 高

---

#### 2. MCPサーバーのコンテナ化不足 ⚠️

**問題**: ほとんどのMCPサーバーがローカルで実行されており、コンテナ化されていない

**現状**:
- MCPサーバーはPythonモジュールとしてローカル実行
- `konoha_mcp_servers/manaos_unified_system_mcp`のみコンテナ化済み
- 環境依存の問題

**影響**:
- 環境の違いによる動作不良
- デプロイの困難
- スケーラビリティの低下

**対策**:
1. MCPサーバーをコンテナ化
2. Docker Composeで管理
3. 環境変数で設定を管理

**優先度**: 🔴 中

---

#### 3. 接続エラーの多発 ⚠️

**問題**: MCPサーバーの接続エラーに関するドキュメントが多数存在

**現状**:
- `n8n_mcp_server/接続エラー解決手順.md`
- `n8n_mcp_server/接続エラー対処法.md`
- その他、エラー関連ドキュメントが多数

**影響**:
- ユーザー体験の低下
- デバッグの困難
- 運用コストの増加

**対策**:
1. 統一エラーハンドリングの実装
2. リトライメカニズムの強化
3. 接続状態の監視
4. エラーログの統一

**優先度**: 🔴 高

---

### 🟡 中程度の問題点

#### 4. Cursor MCP設定の複雑さ ⚠️

**問題**: 複数のMCPサーバーを個別に設定する必要がある

**現状**:
```json
{
  "mcpServers": {
    "n8n": { ... },
    "svi-video": { ... },
    "manaos-unified": { ... }
  }
}
```

**影響**:
- 設定の複雑さ
- メンテナンスの困難
- エラーの発生しやすさ

**対策**:
1. 統合MCPサーバーの使用を推奨
2. 設定スクリプトの提供（`add_to_cursor_mcp.ps1`は既に存在）
3. 設定テンプレートの提供

**優先度**: 🟡 中

---

#### 5. 環境変数の管理 ⚠️

**問題**: 各MCPサーバーで環境変数が個別に管理されている

**現状**:
- 各MCPサーバーで個別に環境変数を設定
- 設定の重複
- 管理の困難

**影響**:
- 設定の不一致
- メンテナンスの困難

**優先度**: 🟡 中

## ✅ 実施済み改善（2026-01-28）

### Phase 1: コンテナ関連の緊急改善

#### 1. リソース制限の追加 ✅

すべてのサービスにリソース制限を追加しました：

- **unified-api**: CPU 2.0 / Memory 2G（制限）、CPU 0.5 / Memory 512M（予約）
- **step-deep-research**: CPU 1.5 / Memory 1.5G（制限）、CPU 0.25 / Memory 256M（予約）
- **gallery-api**: CPU 1.5 / Memory 1.5G（制限）、CPU 0.25 / Memory 256M（予約）
- **llm-routing-api**: CPU 1.5 / Memory 1.5G（制限）、CPU 0.25 / Memory 256M（予約）
- **mcp-api**: CPU 1.5 / Memory 1.5G（制限）、CPU 0.25 / Memory 256M（予約）
- **その他のサービス**: CPU 1.0 / Memory 512M（制限）、CPU 0.25 / Memory 128M（予約）

#### 2. サービス依存関係の明確化 ✅

以下のサービスに`depends_on`を追加：

- **mcp-api**: `unified-api`と`llm-routing-api`に依存（`condition: service_healthy`）
- **gallery-api**: `unified-api`に依存

これにより、依存サービスが正常に起動してからサービスが起動するようになりました。

#### 3. Windowsパスの問題を修正 ✅

**mcp-api**サービスの`OBSIDIAN_VAULT_PATH`環境変数のデフォルト値を修正：

- **修正前**: `C:/Users/mana4/Documents/Obsidian Vault`（Windowsパス直接指定）
- **修正後**: `/app/obsidian_vault`（コンテナ内パス）

環境変数で外部から設定可能なため、Windowsパスが必要な場合は環境変数で指定可能です。

#### 4. ログローテーション設定の追加 ✅

すべてのサービスにログローテーション設定を追加：

```yaml
logging:
  driver: "json-file"
  options:
    max-size: "10m"
    max-file: "3"
```

これにより、各サービスのログファイルが10MBごとにローテーションされ、最大3ファイルまで保持されます。

#### 5. Healthcheckの確認 ✅

主要なDockerfileを確認し、curlが適切にインストールされていることを確認：

- ✅ `unified_api_server/Dockerfile` - curlインストール済み
- ✅ `step_deep_research_service/Dockerfile` - curlインストール済み
- ✅ `gallery_api_server/Dockerfile` - curlインストール済み
- ✅ `mcp_api_server/Dockerfile` - curlインストール済み
